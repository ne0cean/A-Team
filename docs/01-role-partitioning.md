# 01. 역할 분할 원칙

병렬 작업의 핵심은 에이전트들이 서로의 작업을 덮어쓰지 않는 것이다.
역할은 두 축으로 분리한다: **레벨(추상화 수준)** + **도메인(파일 영역)**.

---

## 레벨 분리 (권장 기본 구조)

```
Agent A — 아키텍처 / 신규 기능
  역할: 시스템 설계, 새로운 패턴 도입, 복잡한 비즈니스 로직
  특징: 파일 구조 변경 가능, 새 파일 생성 가능
  모델: Claude Opus / Sonnet (고차원 추론 필요)

Agent B — 최적화 / 클린업
  역할: 코드 품질, 성능, 배포 안정성, 반복 작업
  특징: 기존 파일 내부 수정 위주, 구조 변경 없음
  모델: Gemini Pro / Claude Haiku (반복 처리에 효율적)
```

두 에이전트가 같은 파일을 건드릴 가능성이 있으면 레벨 분리가 실패한 것이다.
태스크 플랜을 작성할 때 이 충돌을 미리 탐지한다.

---

## 도메인 분리 예시

프로젝트 특성에 따라 도메인을 나눈다:

### 풀스택 웹앱
```
Agent A (아키텍처)  →  client/src/pages/, hooks/, store/
Agent B (최적화)    →  server/, client/src/components/, scripts/
```

### 마이크로서비스
```
Agent A  →  service-A/
Agent B  →  service-B/
공유 금지 →  shared/ (한 에이전트만 담당, 변경 시 사전 통보)
```

### 단일 대형 파일
```
한 에이전트만 담당. 분리 불가능한 파일은 직렬 처리.
```

---

## 분리 불가 파일 처리

충돌이 불가피한 파일은 **직렬 처리 블록**으로 명시한다:

```markdown
## 직렬 처리 (순서 엄수)
1. Agent A가 schema.ts 수정 완료 → CURRENT.md에 완료 기록
2. Agent B가 CURRENT.md 확인 후 schema.ts를 기반으로 작업 시작
```

---

## 안티패턴

```
❌ "둘 다 server.js 건드려도 괜찮겠지" — 반드시 충돌
❌ 태스크 플랜 없이 구두로 역할 합의 — 세션 종료 후 맥락 소실
❌ 한 에이전트가 다른 에이전트 영역 리팩토링 — 모르는 사이에 덮어씌움
```
