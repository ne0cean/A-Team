# RFC-001 Prompt Caching

## 1. Problem Statement
현재 A-Team 세션은 매 턴마다 CLAUDE.md + MEMORY.md + `.context/CURRENT.md` + governance rules (총 6–12k tokens)를 full re-read 한다. `scripts/daemon-utils.mjs::callSdkWithAdvisor()`는 Anthropic SDK의 `cache_control`을 사용하지 않아, B1–B3 벤치에서 Session 2+ 재진입 시에도 cold-read 비용을 그대로 지불한다. 세션당 M1(토큰)의 ~40%가 "이미 읽은 불변 컨텍스트"에 소진되며, B3(multi-file refactor)처럼 동일 system prompt를 수십 턴 반복하는 벤치에서 낭비가 가장 크다. P1(thin-wrapper) 아키텍처는 subagent 분기마다 system prompt를 재전송하므로 이 낭비가 9배로 증폭된다.

## 2. Strength Claim
**"Prompt Caching이 P5(`.context/CURRENT.md` 세션 연속성)를 Anthropic-native ephemeral 캐시 레이어 주입 방식으로 강화한다."** — CURRENT.md를 5min TTL, CLAUDE.md + governance를 1h TTL breakpoint로 묶어 Session 2+에서 `cache_read_input_tokens`로 치환. P5의 "연속성" 속성은 보존되고, "비용"만 감소.

## 3. Integration Design

**Surface**: `scripts/daemon-utils.mjs` L45–70

**New function**:
```javascript
// scripts/daemon-utils.mjs
export async function cachedMessageCreate(client, {model, systemPrompt, docs, messages, maxTokens}) {
  const system = [
    { type:"text", text: systemPrompt, cache_control:{type:"ephemeral", ttl:"1h"} }, // CLAUDE.md+rules
    { type:"text", text: docs,         cache_control:{type:"ephemeral"} }            // CURRENT.md 5min
  ];
  const r = await client.messages.create({ model, max_tokens:maxTokens, system, messages });
  logAnalytics({ event:"cache", hit: r.usage.cache_read_input_tokens>0,
                 read: r.usage.cache_read_input_tokens, write: r.usage.cache_creation_input_tokens });
  return r;
}
```

**Hook insertion**: `callSdkWithAdvisor()` L58 — `client.messages.create(...)` 호출을 `cachedMessageCreate(...)`로 교체. `ENABLE_PROMPT_CACHING !== "false"`일 때만 신규 경로.

**New files**:
- `.context/CACHE_MANIFEST.md` — breakpoint 순서 감사 (1h block MUST precede 5min block)
- `governance/rules/prompt-caching.md` — workspace ID 격리 정책
- `.mcp.json` — `"workspace":"ateam-${REPO_HASH}"` 필드 추가

## 4. Implementation Plan

**Phase 1 — Wrapper (day 1)**: `cachedMessageCreate()` + unit test. Analytics event stream to `.context/analytics/cache.jsonl`.

**Phase 2 — Integration (day 2)**: `callSdkWithAdvisor()` 경로 분기. 9개 subagent 시스템 프롬프트에 breakpoint 적용. `.mcp.json` workspace 필드 추가.

**Phase 3 — Audit + rollout (day 3)**: `CACHE_MANIFEST.md` 자동 생성 스크립트 (`scripts/cache-audit.mjs`). B1–B3 A/B 벤치 3회 반복, σ 계산. Gate 판정 후 기본 활성화.

## 6. Rollout + Rollback
- **Flag**: `ENABLE_PROMPT_CACHING` (default `true` after Gate). `.env.example`에 추가.
- **Rollout**: Phase 3 종료 후 main 병합. 1주 모니터링 (`cache_hit_rate >= 0.6` 알람).
- **Rollback**: `export ENABLE_PROMPT_CACHING=false` → 즉시 legacy. Git revert `daemon-utils.mjs` + `.mcp.json` 2-file revert.

## 7. Success Criteria (G5 Mapping)
- **G5-a** (≥15% 1개 메트릭): M1 −35% predicted (DD-01) → PASS
- **G5-b** (≤5% 악화): wrapper overhead <50ms, M2 net 개선 → PASS
- **G5-c** (M4 퇴행 금지): 캐시는 내용 불변 → correctness invariant → PASS
- **G5-d** (B1–B6 중 ≥4개 개선): B1/B2/B3/B5 모두 시스템 프롬프트 재사용 구조 → PASS
- **G5-e** (σ < mean×0.1): 캐시 hit 결정론적, 변동성 낮음 → PASS

# RFC-002 Handoff Compression

## 1. Problem Statement
`scripts/model-exit.sh`의 `/handoff`는 현재 `CURRENT.md`를 그대로(raw 6–10k tokens) 다음 세션의 opening context로 주입한다. 다음 세션 cold-start에서 LLM은 narrative/history/blocker가 뒤섞인 원문을 매번 재파싱하며, M2(wall-clock) 측정 시 첫 45초가 "지난 세션 복원"에 소모된다. P5(Mirror/Handoff)의 **의도는 연속성**이지만, **구현은 비압축 복사**에 머무르고 있다. B4(UI+시각)·B5(research) 처럼 긴 세션 후 재개하는 벤치에서 이 비용이 누적된다.

## 2. Strength Claim
**"Handoff Compression 5-layer가 P5(세션 연속성)를 의미 보존 레이어 분리(Facts/Story/Reasoning/Action/Caution) 방식으로 강화한다."** — raw dump을 5-레이어 구조로 치환해 다음 세션이 O(1) lookup 가능. P5의 "상태 이전" 계약은 유지되고, 표현 밀도만 74% 향상.

## 3. Integration Design

**Surface**: `lib/handoff-compressor.ts` (신규) + `scripts/model-exit.sh` L20–45 (기존).

**New module**:
```typescript
// lib/handoff-compressor.ts
export interface Handoff5Layer { facts:string; story:string; reasoning:string; action:string; caution:string; }
export function compress5Layer(currentMd: string): Handoff5Layer { /* regex+Haiku extract */ }
export function render(layer: Handoff5Layer): string; // -> HANDOFF_PROMPT.txt
```

**Hook insertion**: `model-exit.sh` L28 `cp CURRENT.md HANDOFF.txt` → `node scripts/run-compressor.mjs`. 출력 경로 `.context/HANDOFF_PROMPT.txt` (기존 `/session-start` 훅이 우선 읽음).

**Archival**: 매 handoff마다 `.context/archive/CURRENT-${SESSION_ID}.md` 풀본 보존 (손실 복구용).

**New files**: `lib/handoff-compressor.ts`, `scripts/run-compressor.mjs`, `governance/rules/handoff-compression.md` (5-layer 경계 예시 4–5개), `.context/archive/` 디렉터리.

## 4. Implementation Plan

**Phase 1 — Compressor core (day 1–2)**: 5 extractor 함수(regex-first, Haiku fallback). `compress5Layer()` + render. 최소 1.2k tokens floor 로직.

**Phase 2 — Integration (day 3)**: `model-exit.sh` 수정, archive 자동 생성, `--full-handoff` escape hatch 플래그.

**Phase 3 — Governance + A/B (day 4)**: 5-layer 경계 예시 문서. B4/B5 A/B 벤치 3회.

## 6. Rollout + Rollback
- **Flag**: `COMPRESSION_MODE` = `off` | `on` | `shadow` (shadow = 둘 다 생성, 비교만). Default `shadow` 1주 → `on`.
- **Rollout**: Phase 3 Gate 통과 후 `on`. archive 디렉터리 무제한 보존 (사용자 수동 prune).
- **Rollback**: `export COMPRESSION_MODE=off` → `cp CURRENT.md HANDOFF_PROMPT.txt` 경로로 즉시 복귀. `lib/handoff-compressor.ts`는 남겨두고 호출만 차단.

## 7. Success Criteria (G5 Mapping)
- **G5-a**: M1 −74% on handoff payload → PASS
- **G5-b**: 압축 runtime ≤2s (Haiku), M2 net −60%+ → PASS
- **G5-c**: Facts/Caution layer는 lossless (테스트 강제), M4 invariant → PASS
- **G5-d**: B4/B5/B6 세션 재개 포함 벤치 + B3 long-refactor → 4개 개선 → PASS
- **G5-e**: regex-first 결정론, Haiku 부분만 σ 발생 → 3회 반복 σ<10% 예상 → PASS
