# /investigate — 체계적 근본 원인 분석

**핵심 원칙**: 근본 원인 없이 수정 금지. 증상 처치는 임시방편이다.

## 언제 사용하나
- 버그/에러의 원인을 파악해야 할 때
- "왜 이게 안 되지?" 상황
- 재현은 되는데 원인을 모를 때
- researcher가 아닌 이 스킬을 사용할 것 (researcher는 정보 수집, investigate는 인과 분석)

---

## Phase 1: 근본 원인 조사

**증상 수집**
```bash
# 에러 메시지, 스택 트레이스 전체 읽기
# 최근 변경사항 확인
git log --oneline -10
git diff HEAD~3..HEAD -- [관련 파일]
```

**재현 확인**
- 증상을 정확히 재현하는 최소 케이스 찾기
- 재현 불가 시: NEEDS_CONTEXT 반환 (재현 조건 명시)

**출력**: 테스트 가능한 근본 원인 가설 1개

---

## Phase 2: 패턴 매칭

알려진 버그 패턴과 대조:
- 경쟁 조건 (race condition)
- Null/undefined 전파
- 상태 오염 (state corruption)
- 외부 서비스 통합 실패
- 설정 drift
- 오래된 캐시

패턴 매칭 실패 시 → 외부 검색 (민감 정보 제거 후)

---

## Phase 3: 가설 검증

**가설 테스트 절차**
1. 가설을 반증할 수 있는 디버그 출력 추가
2. 테스트 실행 또는 재현
3. 결과로 가설 확인/반증

**실패 한도**: 가설 3개 연속 반증 시 → BLOCKED 반환
- 이유: 아키텍처 수준 문제일 가능성. 더 이상 추측 금지.
- BLOCKED 메시지에 시도한 가설 3개와 반증 근거 포함

---

## Phase 4: 수정 & 검증

**수정 원칙**
- 증상이 아닌 근본 원인만 수정
- diff는 최소화 (관련 없는 변경 포함 금지)
- 수정 범위가 5개 파일 초과 시 → AskUserQuestion으로 확인 후 진행

**회귀 테스트 필수**
```
수정 전: 테스트 실패 확인
수정 후: 테스트 통과 확인
전체 테스트 스위트: 사이드 이펙트 없음 확인
```

---

## Phase 5: 보고서

```json
{
  "status": "DONE | DONE_WITH_CONCERNS | BLOCKED",
  "symptom": "[관찰된 증상]",
  "root_cause": "[검증된 근본 원인]",
  "fix": {
    "files_modified": ["[파일 경로]"],
    "description": "[수정 내용]"
  },
  "evidence": "[수정 전 실패 / 수정 후 통과 증거]",
  "regression_test": "[추가한 회귀 테스트]",
  "risks": ["[남은 위험 요소]"]
}
```

## 스코프 잠금 (선택)
특정 디렉토리로 수정 범위 제한 가능:
`/investigate --scope src/auth/` → auth 디렉토리 외 수정 금지
