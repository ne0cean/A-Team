## 스타트업 모드 (A)

6가지 핵심 질문을 하나씩 순서대로 (한 번에 여러 개 금지):

1. **수요 현실**: "지금 이 문제로 고통받는 사람을 3명 이상 직접 인터뷰했나요? 뭐라고 했나요?"
2. **현재 해결책**: "지금 이 사람들은 어떻게 해결하고 있나요? 왜 그게 부족한가요?"
3. **절박한 특수성**: "이 제품이 없으면 당장 내일 큰 손실이 생기는 사람은 누구인가요?"
4. **가장 좁은 진입점**: "전체 비전의 1%만으로 검증할 수 있는 가장 작은 것은 무엇인가요?"
5. **직접 관찰**: "이 문제를 직접 겪어봤나요? 언제, 어떻게요?"
6. **미래 적합성**: "5년 후 이 시장은 어떻게 변해있을까요? 그 변화가 이 제품에 유리한가요?"

답변 후 → 전제 도전 → 대안 강제 생성 → 설계 문서 작성

## 빌더 모드 (B)

1. **맥락 파악**: 사용자의 아이디어를 1-2문장으로 재해석한 뒤, **단 1개 질문**으로 핵심 맥락을 파악한다. 질문은 "누구를 위해"에 집중.
   예: "[아이디어 요약]. 이걸 **본인이 매일 직접 쓰기 위해** 만드시나요, 아니면 **다른 사람도 쓸 수 있도록** 만드는 건가요?"
2. **랜드스케이프 검색**: 유사한 도구/라이브러리 웹 검색
3. **3가지 접근법 강제 생성**: 서로 다른 방향의 대안 3개
4. **시각적 스케치**: 텍스트 기반 와이어프레임 또는 아키텍처 다이어그램
5. **전제 도전**: "이 가정이 틀렸다면?"

## Amazon 모드 (C) — Working Backwards PR/FAQ

제품이 이미 출시됐다고 가정하고, 보도자료(PR)와 FAQ를 먼저 작성하여 고객 가치를 역방향으로 검증한다.

### Step 1: PR (Press Release) 작성 — 1페이지 이내

사용자와 대화하며 다음 항목을 채운다 (하나씩 순차):

1. **제목**: 고객 관점의 헤드라인 (기술 용어 금지)
2. **부제**: 핵심 가치를 한 문장으로
3. **1단락 요약**: 누구를 위해, 무슨 문제를, 어떻게 해결하는가
4. **고객 인용**: 가상의 이상적 고객이 말하는 1-2문장
5. **핵심 기능 3개**: 각 기능이 고객에게 주는 구체적 이점

### Step 2: FAQ 작성

**고객 FAQ 3개** (외부 시선):
- "기존 [대안]과 뭐가 다른가요?"
- "얼마인가요? / 무료인가요?"
- "[핵심 우려사항]은 괜찮나요?"

**내부 FAQ 3개** (팀/투자자 시선):
- "시장 규모는?"
- "왜 지금인가?"
- "가장 큰 기술적 리스크는?"

각 답변은 2-3줄. 답할 수 없으면 `_TBD_`로 표시.

### Step 3: PR/FAQ 기반 판정

| 판정 | 조건 | 다음 단계 |
|------|------|-----------|
| **진행** | PR이 설득력 있고 FAQ에 치명적 `_TBD_` 없음 | Phase 0.5 유형 감지로 |
| **보류** | 고객 인용이 억지스럽거나 핵심 FAQ가 `_TBD_` | PR 재작성 또는 중단 제안 |
| **재정의** | FAQ 과정에서 더 나은 방향 발견 | 재정의된 PR로 재시작 |

## 설계 문서 저장

대화 완료 후 자동 저장:
```
~/.gstack/projects/[프로젝트명]-[날짜].md
```

포함 내용:
- 핵심 전제와 검증 상태
- 선택한 접근법과 이유
- 포기한 대안과 이유
- 다음 단계 (코드 작성 전 해야 할 것)

---

## 원칙
- 코드 작성 금지 (이 스킬은 발견과 검증만)
- 질문은 하나씩 — 여러 질문을 한꺼번에 던지지 않음
- 항상 2-3개 대안 생성 (단일 경로 추천 금지)
- 승인 없이 다음 단계 진행 금지
