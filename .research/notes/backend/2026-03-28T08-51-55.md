## 확장성 분석

### 수평 확장(멀티 노드) 시 문제점

**[블로커 1] io.sockets.sockets.get(targetId) — 로컬 소켓 레지스트리 직접 접근**
```js
// line 522, 549
const targetSock = io.sockets.sockets.get(targetId);
if (!targetSock?.connected) { ... sendPush ... }
```
Redis adapter 환경에서는 **다른 노드에 연결된 소켓을 찾을 수 없음**. push 알림 로직이 단일 노드에서만 동작.
→ `io.fetchSockets({ rooms: [targetId] })` 또는 별도 Redis presence key로 대체 필요.

**[블로커 2] 모든 In-Memory Store — 노드 간 공유 불가**
| Store | 위치 | 공유 방법 |
|---|---|---|
| `gpsUsers` | Object | Redis Hash + GEORADIUS |
| `privateRooms` | Object | Redis Hash |
| `activeDmSessions` | Map | Redis Set |
| `pendingDmRequests` | Map | Redis Hash + TTL |
| `roomSettingsMap` | Map | Redis Hash |
| `ipConnectionCount` | Map | Redis Counter |
| `rateLimitMap` | Map | Redis + INCR/EXPIRE |

**[블로커 3] GPS 근접 계산 — 공간 인덱스 없음**
현재: O(N) 브루트포스 haversine 계산
→ Redis GEOADD / GEORADIUS 또는 Geohash grid로 O(log N) 대체 가능.
수평 확장 시 각 노드가 부분 gpsUsers만 보유 → 분산 근접 계산 불가.

**[추가 사항] Socket.io Redis Adapter 도입 시 변경 최소 코드**
- `io.to(userId).emit()` 패턴 — adapter 교체 후 그대로 동작 ✅
- `io.engine.clientsCount` — 로컬 카운트만 반영 → 전체 클러스터 카운트는 별도 Redis counter 필요
- `socket.join(roomId)` / `socket.to(roomId)` — adapter가 처리하므로 코드 변경 불필요 ✅
