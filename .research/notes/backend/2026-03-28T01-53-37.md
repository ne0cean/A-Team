### 확장성

**핵심 문제: 전체 상태 단일 프로세스 인메모리**

line 111-127에서 정의된 모든 상태 저장소:
```
gpsUsers{}              → GPS 위치 데이터 (전체 유저)
privateRooms{}          → 룸 상태
activeDmSessions Map    → DM 세션
pendingDmRequests Map   → DM 요청 대기열
rateLimitMap            → Rate limit 카운터
ipConnectionCount       → IP별 연결 수
pushSubscriptions       → 푸시 구독 정보
```
→ **2개 이상 인스턴스 시 유저가 서로 다른 프로세스에 분리 → GPS 근접 탐지 불가, DM 불가**

**Redis adapter 도입 시 변경이 필요한 코드 위치:**
1. `gpsUsers` (line 111) → Redis HSET (GPS 좌표), GEOADD (공간 인덱스)
2. `privateRooms` (line 112) → Redis Hash (룸 메타) + Socket.io room broadcast는 adapter가 처리
3. `rateLimitMap` (line 121) → Redis INCR + EXPIRE (TTL 기반 자동 만료)
4. `pendingDmRequests` (line 119) → Redis HSET with TTL
5. `activeDmSessions` (line 118) → Redis Set
6. `ipConnectionCount` (line 122) → Redis INCR (분산 IP 제한)

**GPS O(n²) 알고리즘 병목** (line 203-210, 302-341)
```js
// gps_update 핸들러에서 매번:
const prevNearby = getNearbyUsers(socket.id)  // O(n) - 전체 유저 순회
// ... 위치 업데이트 ...
const currNearby = getNearbyUsers(socket.id)  // O(n) - 또 전체 순회
```
- 1000명 × 초당 1회 업데이트 = **백만 회 haversine 계산/초**
- 수평 확장해도 각 노드의 gpsUsers가 분리되어 있어 근접 탐지 자체가 불가능
