### 안정성 — 이슈

- **⚠️ 메모리 누수: httpRateLimitMap 정리 주기 과김** (line 45-61)
  `setInterval`로 5분마다 정리, TTL은 1분. 고트래픽 시 Map 크기 지수 증가 위험.
  예: 초당 100 req × 5분 미정리 = 최대 30,000개 엔트리 동시 보유 가능.

- **⚠️ 메모리 누수: rateLimitMap 만료 미처리** (line 177-193)
  `clearRateLimit()`은 disconnect 시에만 호출됨. zombie 연결(disconnect 미발화)의 rate limit 데이터는 영구 잔존.

- **⚠️ 메모리 누수: pushSubscriptions** (line 110-127, 149)
  만료(30분) 설정은 있으나 정리는 5분 간격 주기로만 수행. 서버 재시작 전까지 데이터 누적.

- **⚠️ Privacy tier 4 좌표 노출** (line 331-340)
  `publicUser`로 privacy 필터링하지만 근처 사용자에게 원본 `user.x/user.y` 좌표를 전송.
  tier 4(최고 보안) 사용자는 위치를 숨겨야 하는데 좌표가 그대로 전송됨 — **보안 이슈**.

- **⚠️ 프라이빗 룸 중복 입장 시 에러 응답 없음** (line 471-472)
  같은 사용자가 두 번 `join_space`할 때 `filter`로 중복 제거만 하고 클라이언트에 에러 미전달.
  클라이언트가 입장 실패를 알 수 없음.

- **⚠️ isSameSpace() race condition 가능성** (line 489-507)
  프라이빗 룸 사용자는 `socket.data.roomId`에 의존. 모드 전환 중 race condition 발생 시 데이터 불일치 가능.

### 확장성 이슈

- **❌ Redis Adapter 미지원 — 수평 확장 불가능**
  현재 모든 상태(`gpsUsers`, `privateRooms`, `activeDmSessions`, `pendingDmRequests`, `roomSettingsMap`)가 단일 프로세스 메모리에 존재.
  멀티 인스턴스 배포 시 각 서버가 서로 다른 상태를 가지게 되어 룸 간 통신 불가.

- **Redis 도입 시 수정 필요 지점:**
  1. `const io = new Server(...)` → `io.adapter(createAdapter(pubClient, subClient))` 추가
  2. `gpsUsers`, `privateRooms` 등 모든 in-memory store → Redis Hash/Set으로 마이그레이션
  3. `roomSettingsMap` → Redis Hash로 교체
  4. rate limit 로직 → Redis + TTL 기반으로 변경 (메모리 누수 문제도 동시 해결)
  5. `pushSubscriptions` → Redis로 이동 후 TTL 자동 만료 활용

- **⚠️ 클러스터 모드 미지원**
  Node.js cluster 모듈 또는 PM2 cluster 사용 시 동일한 in-memory 공유 불가 문제 발생.
  Redis adapter 도입 전까지 단일 프로세스 운영 필수.
