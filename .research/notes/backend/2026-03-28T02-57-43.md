**상태 저장소 전체 목록** (line 111-127):

httpRateLimitMap        → HTTP 엔드포인트 레이트 리밋

**수평 확장 시 문제점**:
1. **GPS 근접 탐지 불가** — 두 유저가 다른 노드에 있으면 서로의 gpsUsers에 등록되지 않음 → getNearbyUsers() 결과 불완전
2. **DM 크로스노드 불가** — `io.to(targetId)` 는 같은 노드의 소켓에만 전달 (redis adapter 없이)
3. **룸 브로드캐스트 분리** — 같은 roomId에 접속한 유저가 다른 노드에 있으면 socket.to(roomId) 미전달
4. **IP 제한 우회** — ipConnectionCount가 노드별로 별도 → MAX_PER_IP=5가 실제로 nodes×5로 늘어남
5. **Demo Bot 분리** — 각 노드의 봇들이 DEMO00 룸에 접속하지만 서로 다른 노드에 격리됨

**GPS O(n²) 알고리즘** (line 203-210, 302-341):
```js
// gps_update 이벤트마다:
const prevNearby = new Set(getNearbyUsers(socket.id).map(...))  // O(n) haversine 전체 순회
user.lat = lat; user.lng = lng;
const currNearby = getNearbyUsers(socket.id)  // O(n) 재순회
```
- 유저 1000명, 초당 1회 gps_update = 2,000,000 haversine 계산/초
- update_privacy, audio_mute, move, shout, ping_wave, send_reaction 에서도 각각 getNearbyUsers() 호출
- 동시다발적 이벤트 시 이벤트 루프 블로킹 위험

## 단계 4 — 개선 제안 (우선순위순)

### [높음] 1. socket.data.id → socket.id 직접 사용
- **수정 위치**: `server.js:735`
- **현재**: `const { roomId, id, mode } = socket.data || {};` → `u.id !== id`
- **수정**: `const id = socket.id; const { roomId, mode } = socket.data || {};`
- **영향**: disconnect 핸들러에서 좀비 유저 데이터 잔존 방지
- **난이도**: 낮음 (1줄 수정)
- **관련**: update_privacy (line 353), audio_mute (line 592), ping_wave (line 651) 등도 `id`를 `socket.data.id`에서 가져오므로 동일 패턴 점검 필요

### [높음] 2. GPS Geohash 버킷팅 — O(n) → O(k) 근접 탐색
- **수정 위치**: `server.js:203-210` (getNearbyUsers 함수)
- **방향**:
  - 유저를 geohash 정밀도 6 셀에 버킷팅 (대략 1.2km × 0.6km 셀)
  - `gpsUsers` 외에 `geoHashBuckets: Map<string, Set<socketId>>` 추가
  - getNearbyUsers() → 인접 9개 셀 유저만 haversine 계산
  - gps_update 시 셀 변경 감지 → 버킷 이동
- **예상 효과**: 도시 밀집 시 100m 반경 내 유저 비율 1% 가정 시 99% 계산량 감소
- **난이도**: 중간 (geohash 라이브러리 `ngeohash` 추가, getNearbyUsers 로직 교체)

### [높음] 3. Redis Adapter + 분산 상태 스토어
- **수정 위치**: `server.js:111-127` 전체 상태 초기화 구간 + Socket.io 초기화
- **패키지**: `@socket.io/redis-adapter`, `ioredis`
- **단계적 접근**:
  1. socket.io redis-adapter 적용 → broadcast 분산 (roomId, targetId 기반 이벤트)
  2. rateLimitMap, ipConnectionCount, pendingDmRequests → Redis (단순 INCR/HSET)
  3. gpsUsers → Redis GEO (GEOADD / GEORADIUS) + getNearbyUsers() 교체
  4. pushSubscriptions, activeDmSessions → Redis Hash with TTL
- **난이도**: 높음 (아키텍처 전체 변경, 테스트 환경 Redis 필요)
