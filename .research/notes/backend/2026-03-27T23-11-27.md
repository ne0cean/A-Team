### ⚠️ 안정성 이슈

#### ISSUE-1: disconnect 핸들링 — `socket.data.id` vs `socket.id` 불일치 위험 (중간)
- 라인 735: `(u) => u.id !== id` — 여기서 `id`는 `socket.data.id`
- `socket.data`는 join_space 시 `{ id: socket.id }` 로 세팅되지만, GPS→private 전환 시 덮어씌워지므로 이론상 일치
- 그러나 만약 `socket.data` 세팅 전에 disconnect되면 `id`가 undefined → 필터가 아무것도 지우지 않음 → **좀비 유저 데이터 잔존**
- 라인 734에서 `u.id !== id` 대신 `u.id !== socket.id`를 쓰는 게 안전

#### ISSUE-2: DM 세션 & pending 요청 정리 — 문자열 포함 검사 위험 (낮음)
- 라인 716–721: `key.includes(socket.id)` 로 DM 세션 정리
- socket.id가 다른 socket.id의 부분 문자열일 가능성은 매우 낮지만, 이론적으로 오탐 가능
- 더 안전한 방식: `key.startsWith(socket.id + ':') || key.endsWith(':' + socket.id)`

#### ISSUE-3: `httpRateLimitMap` 메모리 누수 (낮음)
- 라인 55–61: 5분마다 1분 이상 된 항목 정리 — **괜찮음**
- 다만 `rateLimitMap`(socket rate limit)은 disconnect 시 `clearRateLimit()`로 정리되나, clearRateLimit은 O(n) full scan (라인 189–193)
- 동시 연결 1000개 × 이벤트종류 ≈ 수천 개 엔트리 순회 → 잦은 disconnect 시 CPU 부하 가능

#### ISSUE-4: `gps_update` O(n²) 복잡도 위험 (높음)
- 라인 302–341: `gps_update` 마다 `getNearbyUsers()` 두 번 호출
- `getNearbyUsers()`는 `Object.values(gpsUsers)` 전체 순회 → O(n) per call
- GPS 유저 1000명, 업데이트 1초당 1회(rate limit) = 초당 O(n²) 연산
- **1000명 × 1000명 = 백만 번 haversine 계산/초** — 확장 한계의 핵심 병목

#### ISSUE-5: `ping_wave` 이벤트 — 중복 `.find()` 호출 (낮음)
- 라인 662: `privateRooms[roomId].users.find(...)` 를 동일 객체에 두 번 호출 (x, y 각각)
- 미미한 비효율, 가독성 문제

#### ISSUE-6: `isSameSpace()` 클로저 의존 (중간)
- 라인 490–500: `isSameSpace` 함수가 외부 `socket` 변수를 클로저로 캡처
- 내부에서 `socket.data`를 읽어 private 룸 검사 — 이는 정상 패턴이지만
- GPS 모드에서도 private 룸 체크가 필요한 케이스(transition 중)에 false 반환 가능

#### ISSUE-7: `safe()` 래퍼 — 에러 로깅만 하고 클라이언트에 에러 응답 없음 (낮음)
- 라인 197–199: `catch (e) { console.error(...) }` — 클라이언트는 에러를 알 수 없음
- 서버 사이드 오류 디버깅은 가능하나, 클라이언트 UX 처리 불가

## 확장성 분석 (단계 3)

### 현재 수평 확장 시 문제점

#### SCALABILITY-1: 모든 상태가 In-memory (블로커)
- `gpsUsers`, `privateRooms`, `activeDmSessions`, `pendingDmRequests`, `rateLimitMap`, `ipConnectionCount`, `pushSubscriptions` — 전부 단일 프로세스 메모리
- **2개 이상의 Node 인스턴스 = 유저가 서로 다른 인스턴스에 있으면 같은 GPS 공간에서 보이지 않음**
- Socket.io의 `io.to(userId).emit()`이 동작하려면 동일 프로세스에 있어야 함

#### SCALABILITY-2: Redis Adapter 도입 시 변경 필요 위치
1. **라인 31–37**: `new Server(server, {...})` → `@socket.io/redis-adapter` 추가
   ```
   io.adapter(createAdapter(pubClient, subClient));
   ```
2. **라인 111–124**: in-memory 스토어 (`gpsUsers`, `privateRooms` 등) → Redis Hash/Set으로 교체
3. **라인 177–193**: `rateLimitMap` → Redis + EXPIRE TTL로 교체
4. **라인 45–61**: `httpRateLimitMap` → Redis로 교체 (다중 인스턴스에서 IP rate limit 공유 필요)
5. **라인 125–149**: `pushSubscriptions` → Redis Hash + EXPIRE로 교체
6. **라인 203–210**: `getNearbyUsers()` — Redis GEO 명령어 (`GEORADIUS`)로 교체 필요
7. **라인 490–500**: `isSameSpace()` — Redis에서 룸 멤버 조회로 변경 필요

#### SCALABILITY-3: GPS 근접 계산 아키텍처 한계 (핵심)
- 현재: 모든 GPS 유저를 메모리에서 선형 탐색 → O(n) per call
- Redis GEO로 전환해도 네트워크 왕복 비용 발생
- 확장 솔루션: **Geohash 기반 파티셔닝** or **지리적 샤딩** (구현 난이도 높음)

#### SCALABILITY-4: Socket.io 룸 vs. 커스텀 `privateRooms` 객체 이중 관리
- Socket.io의 `socket.join(roomId)` 룸과 `privateRooms[roomId]` 객체를 별도 관리
- Redis Adapter 도입 시 Socket.io 룸은 Redis로 이전되지만 `privateRooms` 커스텀 객체는 별도 처리 필요
- 라인 444: `socket.join(roomId)` + 라인 446: `privateRooms[roomId] = { users: [] }` — 동기화 필요

## 개선 제안 (우선순위순)

### 1. **[높음]** GPS O(n²) 병목 해소 — Geohash 기반 버킷 캐싱
- **문제**: `gps_update` 시 전체 유저 O(n) 순회 2회
- **수정 방향**: `gpsUsers`를 geohash 버킷(예: 정밀도 6 ≈ 1km 셀)으로 파티셔닝
  - `const buckets = {}` — geohash → userIds Set
  - `getNearbyUsers()` 시 인접 9개 버킷만 탐색
- **수정 파일**: `server.js` 라인 111, 203–210 + utils.js
- **예상 난이도**: 중간 (geohash 라이브러리 추가 필요)
- **효과**: O(n) → O(k) where k = 반경 내 유저 수

### 2. **[높음]** `socket.data.id` 대신 `socket.id` 직접 사용
- **문제**: disconnect 핸들러에서 `id = socket.data.id` 사용 → undefined 위험
- **수정 방향**: 라인 735에서 `u.id !== id` → `u.id !== socket.id`
- **수정 파일**: `server.js` 라인 735
- **예상 난이도**: 낮음 (1줄 수정)

### 3. **[높음]** Redis Adapter + 분산 상태 스토어 준비
- **문제**: 전체 상태가 단일 프로세스에 종속 → 수평 확장 불가
- **수정 방향**:
  1. `@socket.io/redis-adapter` 설치 및 `io.adapter()` 설정 (라인 31 근처)
  2. `gpsUsers` → Redis HSET (socketId → JSON)
  3. `privateRooms` → Redis HSET + SET(멤버 목록)
  4. `rateLimitMap` → Redis SET + EXPIRE
  5. `pushSubscriptions` → Redis HSET + EXPIRE
- **수정 파일**: `server.js` 전반 + `utils.js`
- **예상 난이도**: 높음 (전체 구조 리팩토링)

### 4. **[중간]** DM 세션 키 정리 로직 개선
- **문제**: `key.includes(socket.id)` — 이론적 오탐 가능
- **수정 방향**: 라인 716–721에서 `includes` → `startsWith(id + ':') || endsWith(':' + id)`
- **수정 파일**: `server.js` 라인 716–721
- **예상 난이도**: 낮음

### 5. **[중간]** `clearRateLimit()` O(n) 스캔 최적화
- **문제**: 라인 189–193: 전체 rateLimitMap 순회 → disconnect 폭주 시 CPU 부하
- **수정 방향**: socketId → eventKeys Set 역인덱스 유지
  - `rateLimit()` 호출 시 `socketEventIndex[socketId].add(key)` 추가

- **수정 파일**: `server.js` 라인 177–193
- **예상 난이도**: 낮음

### 6. **[낮음]** `safe()` 래퍼에 에러 응답 추가
- **문제**: 에러 발생 시 클라이언트에 응답 없음
