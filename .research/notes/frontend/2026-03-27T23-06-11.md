## 단계 2 — 컴포넌트 & 훅 분석

### Avatar.jsx — 성능 평가: 양호
- `memo(Avatar)` 적용됨 ✅
- 뷰포트 컬링(`isVisible` 체크) 적용 ✅
- `getFace()` 함수: 렌더마다 SVG JSX 새로 생성. useMemo 없음 — character가 많아지면 부하
- 이중 motion.div 중첩(position outer + bob inner): Framer Motion 컨텍스트 두 배 — 필요한 설계이나 무거움
- `key={user.mood ?? 'default'}` 로 mood 변경 시 remount → 의도된 fade 효과 ✅

### ProximityAura.jsx — 성능 이슈 있음
- `memo` 적용됨 ✅
- **`useWindowSize` 훅**: resize 이벤트마다 state 업데이트 → 모든 resize에서 re-render
- 매 렌더마다 모든 유저 거리 계산 (O(n)) — 유저 수 증가 시 비례적 부하
- ConnectionLine.jsx와 **일관성 없음**: ProximityAura는 useWindowSize 사용, ConnectionLine은 `window.innerWidth` 직접 참조 (resize 반응 없음)

### ConnectionLine.jsx — 경미한 이슈
- `memo` 적용됨 ✅
- `window.innerWidth/Height` 직접 접근 — resize 이후 잘못된 거리 계산 가능
- animated `strokeDashoffset`이 항상 `Infinity` repeat — 화면에 없을 때도 애니메이션 지속

### DmPanel.jsx — 최적화 기회
- **`memo` 미적용** — `activeDm` 상태 변경 시 불필요하게 re-render
- `handleSend`, `handleKeyDown` — useCallback 없음 (DM 패널은 사용 빈도 낮아 체감 영향 적음)

### useCanvasInteraction.js — 핵심 훅 분석
- `proximityScaleMap` useMemo 의존성: `[me?.x, me?.y, users]` — users 배열 참조 변경마다 재계산
  - `updateUserPosition`이 `users.map()`으로 새 배열 반환 → 이동마다 근접도 재계산 트리거
- `clusterOffsets` useMemo도 동일 패턴 (`[users, me?.x, me?.y]`)
- 두 useMemo가 O(n²) cluster 계산 — n명이면 초당 최대 12.5 * 2 * n² 연산

### useStore.js — 설계 평가: 합리적
- flat 단일 Zustand store — 심플하고 유지보수 용이
- `updateUserPosition`: 매 이동마다 `users.map()` 실행 (불가피하지만 배열 copy 발생)
- DM 상태(`activeDm`)와 공간 상태(`users`)가 동일 store → 셀렉터로 isolation 가능 ✅

## 단계 3 — 번들 최적화 분석

### 현재 번들 설정 (vite.config.js)
```
vendor-react: react + react-dom + react-router-dom
vendor-motion: framer-motion
vendor-socket: socket.io-client
```
→ 기본 chunking은 잘 설정됨

### 주요 의존성 크기 (gzip 추정)
| 패키지 | 크기 (approx) | 비고 |
|--------|-------------|------|
| `framer-motion` | ~47KB | 별도 chunk 분리됨 ✅ |
| `socket.io-client` | ~45KB | 별도 chunk 분리됨 ✅ |
| `react` + `react-dom` | ~45KB | 분리됨 ✅ |
| **`html2canvas`** | **~800KB** | **⚠️ 가장 큰 라이브러리** |
| `qrcode.react` | ~30KB | 합리적 |
| `lucide-react` | tree-shakeable | 아이콘 수가 많으면 주의 |

### `html2canvas` 문제
- QrShare 컴포넌트에서 QR 이미지 저장 용도로만 사용으로 보임
- 800KB+ 라이브러리가 초기 번들에 포함 가능성 있음
- QrShare 컴포넌트가 `manualChunks`에 포함되지 않음

### 라우트별 lazy loading 미적용
- 현재: 모든 페이지가 하나의 앱 번들에 포함
- RoomPage, LandingPage, SpacePage 등 라우트 분리 여부 불명 (App.jsx 미확인)
- `React.lazy` + `Suspense`로 RoomPage 분리 시 초기 로드 개선 가능

### `@tailwindcss/postcss` 위치 오류
- `dependencies`에 포함되어 있으나 `devDependencies`에 있어야 함
- 프로덕션 번들 크기에 영향 없지만 배포 환경 혼란 유발 가능
