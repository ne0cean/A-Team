## 단계 2 — 위협 모델 분석

### [높음] IP 스푸핑으로 연결 제한 우회
**위치**: `server.js:224-232` (socket.io 미들웨어), `:66-88` (HTTP 엔드포인트)
```js
const ip = socket.handshake.headers['x-forwarded-for']?.split(',')[0].trim()
  || socket.handshake.address;
```
**문제**: `x-forwarded-for` 헤더를 아무 검증 없이 신뢰. 서버가 리버스 프록시 없이 직접 노출된 경우, 공격자가 헤더를 임의로 설정해 IP 기반 연결 제한(`MAX_CONNECTIONS_PER_IP=5`)과 HTTP rate limit을 완전히 우회 가능.
**영향**: DoS, 무제한 연결, 룸 코드 브루트포스 가속화.

### [높음] 룸 코드 열거 (Room Enumeration)
**위치**: `server.js:74-82` (`GET /api/rooms/:code/count`)
```js
const code = String(req.params.code || '').toUpperCase().slice(0, 10);
const room = privateRooms[code];
res.json({ count: room ? room.users.length : 0 });
```
**문제**: 응답의 `count` 값(0 vs 양수)으로 룸 존재 여부 확인 가능. 룸 코드는 `crypto.randomBytes(3).toString('hex')` = 6자리 hex = 16,777,216 경우의 수.
- IP당 rate limit: 3000ms → IP당 최대 ~20 req/min
- IP 스푸핑 가능 시: 초당 수백 요청 가능 → 현재 활성 룸 전체 탐색 가능
**영향**: 공개하지 않은 Private 룸 발견 후 코드로 입장.

### [중간] 모든 룸 멤버가 룸 설정 변경 가능 (무권한 설정 변경)
**위치**: `server.js:604-616` (`set_room_settings`)
```js
socket.on('set_room_settings', safe(({ name, description }) => {
  const { roomId, mode } = socket.data || {};
  if (mode !== 'private' || !roomId) return;
  // 룸에 있는 모든 유저가 변경 가능 — owner 검증 없음
```
**문제**: 룸 생성자(owner)와 일반 멤버를 구분하지 않음. 악의적 유저가 입장 후 즉시 룸 이름/설명 변조 가능.
**영향**: 소셜 엔지니어링 (예: 룸 이름을 사칭), 분쟁.

### [중간] WebRTC SDP 콘텐츠 미검증
**위치**: `server.js:559-577`
```js
if (!offer || typeof offer.sdp !== 'string') return;
io.to(targetId).emit('webrtc_offer', { fromId: socket.id, offer });
```
**문제**: SDP가 문자열인지만 확인, 실제 SDP 형식 검증 없음. 악의적 SDP는 일부 클라이언트 WebRTC 구현에서 예외/크래시를 유발할 수 있음.
**영향**: 클라이언트 DoS (특정 브라우저/구현 대상), 비정상 시그널링 페이로드 릴레이.

### [중간] 이모지 반응(send_reaction) — 이모지가 아닌 문자열 허용
**위치**: `server.js:695-700`
```js
const safeEmoji = sanitizeStr(emoji, MAX_EMOJI); // MAX_EMOJI=10
io.to(targetId).emit('receive_reaction', { fromId: socket.id, emoji: safeEmoji });
```
**문제**: 실제 이모지인지 검증하지 않음. 임의 10자리 텍스트(예: `<script>`) 전송 가능. 클라이언트에서 `dangerouslySetInnerHTML` 등으로 렌더링 시 XSS.
**영향**: 클라이언트 측 XSS (클라이언트 렌더링 방식에 따라 다름).

### [낮음] push_subscribe — 구독 객체 무검증
**위치**: `server.js:511-514`
```js
socket.on('push_subscribe', safe((subscription) => {
  if (!subscription?.endpoint) return;
  pushSubscriptions.set(socket.id, subscription);
```
**문제**: endpoint 존재 여부만 확인, 실제 유효한 Push 구독인지 검증 없음. 악의적 endpoint가 저장되면 서버가 해당 URL로 POST 요청 전송 (SSRF 잠재적 위험).
**영향**: 내부 네트워크 SSRF (endpoint를 `http://internal-service/...`로 설정 시), 의도치 않은 외부 요청.

### [낮음] 메모리 DoS — move 이벤트 과다
**위치**: `server.js:391` (`move`, rate limit 50ms)
**문제**: `move` 이벤트는 50ms(초당 20회) 허용. 1000개 연결 × 20 events/sec = 초당 최대 20,000 이벤트 처리. 각 이벤트는 GPS 모드에서 `getNearbyUsers()` (O(n) 스캔) 호출.
**영향**: 1000명 접속 시 CPU/메모리 과부하.

const io = new Server(server, {

`.env` 파일 누락 또는 배포 시 환경변수 미설정 가능성 존재.

### 인증/인가 부재
- **GPS 모드**: 아무나 입장, 위치만 있으면 됨 (설계 의도)
- **Private 룸**: 코드만 알면 입장 가능, 룸 비밀번호/초대 없음
- **DM**: 같은 공간에 있으면 누구에게나 DM 요청 가능

**위험 수준**: 서비스 모델 특성상 어느 정도 의도된 설계. 단, Private 룸에 비밀번호/초대 기능 없는 점이 열거 공격과 결합 시 위험.

## 단계 4 — 개선 제안 (우선순위순)

### 즉시 수정 필요

1. **[높음] x-forwarded-for 신뢰 범위 제한**
   - **파일**: `server.js:224-232`, `:66-88`
   - **방법**: 리버스 프록시(nginx, Cloudflare 등) 뒤에서만 실행하는 경우, `trust proxy` 설정으로 제한. 직접 노출 시 `x-forwarded-for` 무시하고 `socket.handshake.address`만 사용.
   ```js
   // Express에서
   app.set('trust proxy', 1); // 신뢰하는 프록시 1홉만
   // 또는 특정 IP 범위만 신뢰
   app.set('trust proxy', '10.0.0.0/8');
   ```
   - **난이도**: 낮음 (배포 환경 파악 후 1줄 수정)

2. **[높음] 룸 열거 방지 — 존재 여부 응답 정규화**
   - **파일**: `server.js:74-82`
   - **방법**: 룸이 없어도 항상 `{ count: 0 }` 반환 (현재와 동일). 단, rate limit을 강화하고 룸 코드를 더 길게 만들기.
   - **룸 코드 강화**: `crypto.randomBytes(4)` → 8자리 hex = 4,294,967,296 경우의 수 (256배 증가)
   - **파일**: `utils.js:15-16`
   - **난이도**: 낮음

3. **[중간] 이모지 화이트리스트 적용**
   - **파일**: `server.js:695-700`
   - **방법**: `VALID_MOODS` 처럼 허용 이모지 Set 정의, 또는 Unicode 이모지 범위 정규식 검증
   ```js

   const safeEmoji = EMOJI_REGEX.test(emoji) ? emoji : null;
   ```
   - **파일**: `server.js:695`, 또는 별도 validator 함수
   - **난이도**: 낮음

4. **[중간] push_subscribe SSRF 방지**

   const url = new URL(subscription.endpoint);
   if (!ALLOWED_PUSH_HOSTS.includes(url.hostname)) return;
