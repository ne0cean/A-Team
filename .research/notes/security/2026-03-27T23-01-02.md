| nickname | sanitizeStr(30) — 길이 자르기 | ⚠️ HTML escape 없음 |

| avatar | VALID_AVATARS whitelist | ✅ 양호 |
| privacyTier | parseInt + clamp(1,4) | ✅ 양호 |
| roomCode | String → toUpperCase → slice(0,10) | ✅ 양호 |
| emoji (reaction) | sanitizeStr(10) — 길이만 | ⚠️ whitelist 미적용 |
| SDP (WebRTC) | typeof string 확인만 | ⚠️ 크기 제한 없음 |
| push subscription | endpoint 존재 여부만 확인 | ❌ SSRF 위험 |
| shout message | sanitizeStr(60) | ⚠️ HTML escape 없음 |

### 2-2. 악의적 사용자가 현재 할 수 있는 것

1. **룸 열거(Room Enumeration)**
   - `/api/rooms/:code/count` API는 코드가 존재하면 인원수 반환, 없으면 0 반환
   - rate limit(3000ms)이 있으나 분산 IP로 우회 가능
   - 6자리 hex(16^6 = 16M) brute-force는 비용이 높지만, 특정 패턴 추측은 가능
   - **DEMO_ROOM 'DEMO00'** 코드가 소스에 하드코딩되어 있어 누구나 입장 가능

2. **이모지/reaction 임의 전송**
   - `send_reaction`의 emoji는 whitelist 없이 10자 제한만
   - 악의적인 Unicode 조합 문자(ZWJ 시퀀스 등) 전송 가능
   - 클라이언트 렌더링 방식에 따라 레이아웃 교란 가능

3. **WebRTC SDP 대용량 페이로드**
   - `webrtc_offer/answer`의 SDP 문자열 크기 제한 없음
   - maxHttpBufferSize(100KB) 전체 한도 내에서 대용량 SDP 전송 가능
   - 타겟 클라이언트에게 거대한 SDP 처리 부담 유발

4. **Nickname/Status XSS 주입**
   - `sanitizeStr`은 길이 제한만 수행, HTML entity escape 없음
   - `<script>`, `<img onerror>` 같은 HTML이 닉네임/상태에 포함 가능
   - 클라이언트가 `innerHTML`으로 렌더링 시 XSS 발생 (클라이언트 코드 미확인)

5. **x-forwarded-for 헤더 스푸핑**
   - IP 기반 rate limit과 connection count가 `x-forwarded-for` 헤더를 신뢰
   - reverse proxy 없이 직접 접속 가능한 환경에서 공격자가 헤더 조작 → IP 제한 우회
   - `MAX_CONNECTIONS_PER_IP` (5)와 HTTP rate limit 모두 우회 가능

6. **push_subscribe SSRF**
   - `push_subscribe` 이벤트는 subscription.endpoint URL을 검증 없이 저장
   - `request_dm` / `send_dm_message` 이벤트 트리거 시 `webpush.sendNotification(sub, ...)` 호출
   - 공격자가 `endpoint: "http://internal-service/admin"` 등 내부 URL 지정 가능
   - SSRF 경로: 악의적 endpoint 등록 → 다른 사용자가 DM 요청 → 서버가 내부 URL로 POST

7. **DM 수락 직후 공간 이탈 후 메시지 전송**
   - `accept_dm`으로 세션 수립 후 `leave` 없이 GPS 모드 전환해도 `activeDmSessions`는 유지
   - 공간을 이탈한 사용자에게 계속 DM 전송 가능 (공간 공유 인가 우회)

8. **set_room_settings 인가 부재**
   - 룸에 입장한 사람이면 누구나 `set_room_settings`로 룸 이름/설명 변경 가능
   - 방장(owner) 개념 없음 — 룸 코드를 아는 모든 사람이 관리자 권한

```javascript
const origin = process.env.CLIENT_URL || 'http://localhost:5173';
```

- **현황**: 단일 origin 허용 방식 — 구조적으로 올바름
- **위험**: `CLIENT_URL` 환경변수 미설정 시 `http://localhost:5173`이 프로덕션에서도 허용
- **배포 체크 누락 시 시나리오**: 프로덕션 서버가 로컬 클라이언트 origin을 허용 → 공격자가 로컬에서 프로덕션 서버에 WebSocket 연결 가능

### 3-2. 메모리 내 개인정보
