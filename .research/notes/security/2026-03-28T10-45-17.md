## 핵심 발견 — 위협 모델

### 1. **[높음] 룸 코드 열거 (Room Enumeration)**
- `generateRoomCode()` = `crypto.randomBytes(3).toString('hex')` → 6자리 hex = 16,777,216 조합
- `GET /api/rooms/:code/count`로 존재하는 룸 확인 가능 (count > 0)
- Rate limit: IP당 3초 간격이지만, 분산 IP로 브루트포스 가능
- **위험**: 비공개 룸을 찾아 무단 참여 가능 (룸 참여에 인증 없음)
- **수정 방향**: roomCode 길이를 8-10자로 증가, 또는 `/api/rooms/:code/count`를 룸 참여자에게만 허용

### 2. **[높음] 정밀 위치 정보 노출 (privacyTier 3+)**
- privacyTier 3 이상이면 정밀 GPS 좌표(lat/lng)가 근처 모든 사용자에게 전송
- 악의적 사용자가 같은 근접 반경(30m)에 들어오면 타겟의 정밀 좌표 수집 가능
- `user_gps_moved` 이벤트로 실시간 추적 가능
- **위험**: 스토킹, 정밀 위치 추적
- **수정 방향**: privacyTier 3에서도 소수점 2자리 근사(≈1.1km→너무 큼, 4자리≈11m가 적절), 정밀 좌표는 서버만 보유

### 3. **[높음] socket.id 기반 사용자 식별 — 타 사용자 이벤트 스푸핑 불가하지만 targetId 추측 가능**
- 모든 이벤트에서 `socket.id`를 서버가 자동 설정 → 스푸핑 불가 ✅
- 그러나 `targetId`는 클라이언트가 보낸 값을 그대로 사용
- `isSameSpace()` 체크로 같은 공간 내 사용자만 대상 가능 → 부분 완화
- 하지만 `user_joined` 이벤트로 다른 사용자의 socket.id를 학습 가능
- **위험**: 같은 룸/근접 범위 내에서 원치 않는 DM 요청, reaction 전송
- **수정 방향**: DM은 privacyTier 4만 허용(현재도 클라이언트에서만 제한), 서버에서도 dmEnabled 체크

### 4. **[중간] CORS 설정 — 프로덕션 위험**
- `origin = process.env.CLIENT_URL || 'http://localhost:5173'` — 환경변수 미설정 시 localhost 허용
- 프로덕션 배포 시 CLIENT_URL을 정확히 설정해야 함
- **위험**: CORS 설정 실수로 임의 오리진에서 API 호출 가능 (Socket.io 포함)
- **수정 방향**: 프로덕션에서 CLIENT_URL 필수화, 미설정 시 에러 throw

### 5. **[중간] WebSocket 이벤트 Flooding**
- 대부분 이벤트에 rate limit 있음 ✅ (예: move 50ms, shout 3s, gps_update 1s)
- **하지만**: `push_subscribe`에 rate limit 없음
- `webrtc_ice`는 50ms — ICE candidate 특성상 적절하지만 flooding 가능
- IP당 커넥션 제한: 5개, 전체 제한: 1000개 ✅
- **위험**: 단일 커넥션에서 push_subscribe 반복 호출로 메모리 낭비
- **수정 방향**: push_subscribe에 rate limit 추가 (10초 간격), webrtc_ice는 요청당 candidate 수 제한

### 6. **[중간] 인메모리 상태 — 메모리 누수 위험**
- `gpsUsers`, `privateRooms`, `activeDmSessions`, `pendingDmRequests` 모두 인메모리
- disconnect 시 정리 코드 있음 ✅
- **하지만**: `pendingDmRequests`에 TTL 없음 — disconnect 없이 서버 재시작 전까지 남아있을 수 있음
- `roomSettingsMap`도 빈 룸 삭제 시 같이 정리됨 ✅
- **위험**: 장기 운영 시 메모리 누수 (비정상 연결 해제 시)
- **수정 방향**: pendingDmRequests에 TTL 추가 (5분), 주기적 정리

### 7. **[중간] 입력 검증 — 대체로 양호하나 일부 누락**
- nickname, status: `sanitizeStr`로 길이 제한 ✅
- lat/lng: `validateLatLng`으로 범위 검증 ✅
- avatar: VALID_AVATARS 화이트리스트 ✅
- emoji(reaction): `sanitizeStr(emoji, 10)` — 유니코드 이모지 화이트리스트 아님
- roomCode: `String(req.params.code).toUpperCase().slice(0, 10)` — 패턴 검증 없음
- **위험**: emoji 필드에 XSS 페이로드 삽입 가능 (클라이언트가 innerHTML 사용 시)
- **수정 방향**: emoji를 화이트리스트로 검증 (VALID_MOODS처럼), roomCode는 `/^[A-Z0-9]{6}$/` 정규식

### 8. **[중간] Push Subscription 검증 부재**
- `push_subscribe` 이벤트에서 `subscription.endpoint` 존재만 체크
- 악의적 사용자가 타인의 push subscription을 등록할 수 있음 (endpoint URL 알면)
- **위험**: 잘못된 push subscription으로 타인에게 알림 전송
- **수정 방향**: subscription 객체 구조 검증 (endpoint URL 형식, keys 존재 여부)

### 9. **[낮음] socket.id 노출**
- `user_joined`, `user_moved` 등에서 socket.id가 그대로 전송
- Socket.io 내부 ID는 예측 가능한 패턴 아님 ✅
- **위험**: socket.id가 세션 식별자로 사용되므로 노출 자체는 의도된 동작
- **개선**: 공개용 ID를 별도 생성 (UUID v4)하여 내부 socket.id 분리 권장

### 10. **[낮음] 로그에 socket.id 일부 출력**
- `console.log(`User ${socket.id.slice(0, 6)} joined GPS space`)` — 첫 6자만 ✅
- 프로덕션 로깅에는 구조화된 로거 (winston/pino) 권장
- **위험**: 낮음, 현재 수준 적절

## 설정 검토

### CORS
- Socket.io: `cors: { origin, methods: ['GET', 'POST'] }` — 단일 오리진 ✅
- Express: `cors({ origin })` — 동일 ✅
- **문제**: 환경변수 미설정 시 localhost fallback → 프로덕션 배포 실수 가능

### 개인정보 저장
- IP: `socket.data._ip`에 저장, disconnect 시 `ipConnectionCount`에서 제거
- 닉네임, GPS 좌표: 인메모리, disconnect 시 제거 ✅
- Push subscription: disconnect 후 30분 보관 → 자동 삭제 ✅
- **문제**: GDPR/개인정보보호법 관점에서 IP 처리 정책 문서화 필요

### 인증/인가
- 완전 부재 — 익명 서비스 특성상 의도적
- **위험**: 룸 설정을 아무나 변경 가능 (`set_room_settings` — 룸 참여자만이지만 생성자 구분 없음)
- **수정 방향**: 룸 생성자를 `host`로 지정, 설정 변경은 host만 허용
