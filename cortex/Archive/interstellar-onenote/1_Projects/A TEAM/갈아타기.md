갈아타기
		
		
	
	
		
			
				
에이전트와 코딩 하다가 새로운 윈도우를 열거나 깃헙에 커밋해두고 다른 컴퓨터에서 작업을 이어가면 갑자기 작업이 열화되는데 이걸 방지 할 방법 없을까?
				
ChatGPT의 말:
				
있어요. 원인은 거의 항상&#160;“컨텍스트 단절 + 환경 불일치 + 목표/규칙 재주입 실패”&#160;3종 세트라서, 그걸&#160;구조적으로 강제하면 열화가 거의 사라집니다. 아래를 그대로 쓰면 됩니다.
				
1) 한 줄로 요약: “세션 핸드오프 패킷”을 repo에 박아두기
				
다른 창/다른 PC로 넘어갈 때마다 에이전트가 잃는 건 “기억”이 아니라 **현재 작업 상태(의도/제약/다음 액션/미해결 이슈)**예요.
				
그래서 그걸&#160;파일로 고정해두면, 어디서 열어도 동일하게 이어집니다.
				
추천 파일 3개(가장 효과 좋음)
				
- ./HANDOFF.md&#160;: 지금 무슨 일을 왜 하는지, 다음에 뭘 하면 되는지
				
- ./DECISIONS.md&#160;: 이미 결정한 규칙/가정(다시 토론하면 열화되는 것)
				
- ./TODO_NOW.md&#160;: “다음 30분” 단위 체크리스트
				
핸드오프 템플릿 (복붙용)
				

				
# HANDOFF (last updated: YYYY-MM-DD HH:MM KST)￼
				
## Goal (1–2 lines)￼- 지금 목표:￼
				
## Current status￼- Done:￼- In progress:￼- Blockers:￼
				
## Next 3 actions (must be concrete)￼1)￼2)￼3)￼
				
## Constraints / must-not-break￼- (예: API는 변경 금지, 성능 목표, 대시보드 점수는 위험점수로 통일 등)￼
				
## Key files￼- path:￼- path:￼
				
## Commands to run￼- (예: uv run pytest -q)￼- (예: npm test)￼
				
## Notes for agent￼- (이 세션에서 중요했던 맥락 5줄 요약)￼
				
이걸 커밋하거나(가장 좋음) 최소한 push 해두면 “다른 컴퓨터”에서도 동일 컨텍스트로 재시작 가능.
				

				
2) “환경 불일치” 방지: 실행/의존성을 한 방에 고정
				
다른 PC에서 열화되는 건 모델만의 문제가 아니라&#160;로컬 환경 차이가 섞입니다. (node/python 버전, env vars, OS 차이, 캐시/빌드 산출물)
				
최소 체크리스트
				
- .tool-versions(asdf) 또는&#160;.nvmrc(node),&#160;pyproject.toml/requirements.txt&#160;고정
				
- Makefile&#160;또는&#160;justfile로 표준 명령 제공
				
- .env.example에 필요한 환경변수 목록 명시
				
추천: “한 줄로 재현되는” 명령 만들기
				
- make dev
				
- make test
				
- make lint
				
이게 없으면 에이전트가 다른 PC에서 “어떤 명령을 돌려야 하는지”부터 흔들리면서 퀄리티가 급락합니다.
				

				
3) 새 창/새 세션에서 바로 쓰는 “리부트 프롬프트”
				
새 윈도우에서 첫 메시지에 이걸 넣으면, 에이전트가 헤매는 시간을 확 줄입니다.
				

				
레포의 HANDOFF.md / DECISIONS.md / TODO_NOW.md를 먼저 읽고,￼1) 현재 목표 2) 현재 상태 3) 다음 3개 액션 4) 리스크를 10줄로 요약해줘.￼그 다음 TODO_NOW.md의 1번부터 실행해.￼
				
(핵심: “먼저 읽고 요약 → 다음 액션” 순서 강제)
				

				
4) 커밋 단위도 바꾸면 더 안 열화됨: “작은 커밋 + 설명형 커밋”
				
다른 PC로 넘어갈 때 “중간 상태”가 애매하면 에이전트가 추론을 과하게 하면서 망가져요.
				
- 커밋은 “빌드/테스트 통과” 단위로 최대한 작게
				
- 커밋 메시지에&#160;왜 했는지&#160;1줄 포함
				
예)
				
- feat: add Credit ROC risk scoring (cap roc_ratio; guard low index ROC)
				
- refactor: unify risk score direction (higher = risk)
				

				
5) (진짜 강력) 작업을 “태스크 카드”로 쪼개기
				
/tasks/2026-01-xx-dashboard-risk-score.md&#160;같은 파일 하나 만들고
				
- 배경
				
- 요구사항
				
- 수용 기준(AC)
				
- 완료 정의(DoD)
				
- 테스트 방법
				
을 적어두면, 어느 컴퓨터에서 열든&#160;작업 품질이 일정해져요.
				

				
지금 당장 추천 액션(3분)
				
- 레포 루트에&#160;HANDOFF.md&#160;만들기(위 템플릿 그대로)
				
- DECISIONS.md에 “점수 방향(높을수록 위험)” 같은 불변 규칙 5개 적기
				
- make dev / make test&#160;같은 표준 명령 하나라도 만들기
